Config — src/config/

Shared plumbing every controller pulls in. Nothing here renders HTML; it's connections, session state, and small helper functions reused across modules.

db.phpdatabase connection

Opens the PDO connection to the career_system MySQL database and exposes it as $pdo. Sets ERRMODE_EXCEPTION so a bad query throws instead of failing silently, and FETCH_ASSOC so every query result comes back as a plain associative array. If the connection itself fails, it stops the request with a 500 rather than letting a null $pdo cause confusing errors downstream.

session.phpsession bootstrap, access control, CSRF

Starts (or resumes) the PHP session, then defines the four functions every controller relies on for security:

  • require_login() — redirects to the login page if nobody's signed in.
  • require_role($role) — calls require_login() first, then returns 403 Forbidden if the signed-in user's role doesn't match. This is the line that keeps students out of admin pages and vice versa.
  • csrf_token() / csrf_field() — generates a random 32-byte token per session and renders it as a hidden form field.
  • require_csrf() — checks the submitted token against the session token with hash_equals() (constant-time comparison, avoids timing attacks) and returns 403 if it doesn't match. Every controller that handles a POST calls this before touching the database.
ml_client.phpHTTP bridge to the Flask ML service

PHP has no idea how the AI model works — it just makes two cURL calls to the Python service on port 5001:

  • get_recommendation($payload) — POSTs scores + interest to /predict, returns the decoded JSON (pathway, confidence, explanation, ranking) or an ['error' => ...] array if the service is unreachable or rejects the input.
  • trigger_retrain() — POSTs to /retrain with a 120-second timeout (retraining three models on ~3,000+ rows takes longer than a normal request).

Both functions fail soft: if the ML service is down, the caller gets back an error message to show the user rather than a crash. In student_controller.php this specifically means the student's scores are still saved even if the recommendation step fails.

helpers.phpsmall shared utilities
  • pathway_badge_class($pathway) — maps a pathway name to a CSS class (badge-stem, badge-social, badge-arts) so the colored pill renders consistently everywhere a pathway is shown.
  • validate_password_strength($password, $label) — enforces the 8-char minimum, uppercase, lowercase, digit, and special-character rules. Shared by registration, admin password reset, and profile password change so the rule only lives in one place.
  • require_student_by_id($pdo, $id) — looks up a student by ?id= and 404s if it's not a real student account. Used by every admin action that targets one specific student (delete, reset password).
  • generate_student_id($pdo) — takes the highest existing numeric suffix on a STU% ID and adds one, producing STU001, STU002, etc. Uses MAX() rather than COUNT(*) deliberately: counting rows would generate a duplicate ID once a student has been deleted and left a gap in the sequence.
constants.phpshared value lists

Defines SUBJECTS (the five score fields, each mapped to its display label) and ALLOWED_INTERESTS (the six valid interest categories). Both the score-validation loop in student_controller.php and the form fields in student_dashboard.php read from these same arrays, so a new subject or interest only needs to be added in one place.

Controllers — src/controllers/

Every controller is required at the top of its matching page in public/, runs before any HTML is output, and leaves behind plain PHP variables ($error, $result, $history, …) for the view to print. None of them echo HTML themselves.

login_controller.phpFR1 — authentication

On POST: looks up the email, then password_verify()s the submitted password against the stored bcrypt hash. Three outcomes — no such email, wrong password, or success — each with a distinct, specific error message rather than a generic "invalid credentials", which is a deliberate usability choice (and is exactly what test_web_flow.py asserts on). On success it writes user_id, student_id, full_name, and role into $_SESSION and redirects to the admin or student dashboard based on role.

register_controller.phpFR1 — account creation

Validates all fields are present, the email is well-formed, the password passes validate_password_strength(), and the two password fields match. Checks the email isn't already registered, then hashes the password with password_hash() (bcrypt), generates a student ID via generate_student_id(), inserts the new row, and logs the student straight in — no separate "verify your email" step.

student_controller.phpFR2–FR6 — the core recommendation flow

This is the file the supervisor asked to see the "two-insert flow" in. On POST, after validating every score is a number 0–100 and the interest is one of ALLOWED_INTERESTS, it runs a strict three-step sequence:

1. INSERT INTO academic_records (...)      -- save the raw submission first
   $record_id = $pdo->lastInsertId();

2. $prediction = get_recommendation($ml_payload);   -- HTTP call to Flask :5001

3. INSERT INTO recommendations (user_id, record_id, pathway_id,
                                 confidence, explanation, model_used)
   VALUES (?, ?, ?, ?, ?, ?);

The order matters: scores are saved before the ML call, so if the Flask service is down the student's data isn't lost — they just see "recommendation engine temporarily unavailable, your scores were saved". The record_id from step 1 is what links a recommendation back to the exact scores that produced it (see the schema relationships below). At the bottom of the file, a separate query pulls the student's full recommendation history (joining recommendations + pathways + academic_records) for display, regardless of whether this request was a GET or a fresh POST.

student_profile_controller.phpself-service profile edit

Lets a student update their own name, email, and optionally password. Email uniqueness is checked against every other account (WHERE email = ? AND user_id != ?). A password change only happens if either password field was filled in, and requires the current password to verify first — changing name/email alone never touches password_hash.

admin_controller.phpFR7/FR8 — dashboard data + retrain trigger

Powers the admin dashboard. Four separate queries build the page: overall totals (student count, recommendation count), a pathway breakdown, the 50 most recent recommendations, and the full student roster with a submission count per student (a LEFT JOIN so students with zero submissions still show up). A student's latest pathway is computed in PHP rather than SQL — the query pulls every recommendation ordered newest-first, and a simple loop keeps only the first (most recent) row per user_id, which sidesteps a fragile correlated subquery for "latest row per group" in MySQL. If the retrain button was submitted, it calls trigger_retrain() before any of that, so a fresh retrain result can display above the stats.

admin_student_report_controller.phpFR7 — per-student printable report

Validates the ?id= query param is a real integer, looks up the student, 404s if not found, then pulls their full recommendation history the same way student_controller.php does for a student viewing their own history.

admin_pathway_report_controller.phpFR7 — per-pathway printable report

Looks up the pathway by name from ?pathway=, 404s if it doesn't exist, then pulls every recommendation for that pathway across all students, newest first.

admin_reset_password_controller.phpadmin resets a student's password

Same strength validation as registration, hashes the new password, and updates that one row. No email is sent — the success message explicitly reminds the admin to share the new password with the student directly, since there's no automated password-reset flow.

admin_delete_student_controller.phppermanent student deletion

Requires the admin to type the student's exact full name into a text field as a confirmation gate before anything happens. On match, it runs a single DELETE FROM users WHERE user_id = ? AND role = 'student' — the role = 'student' guard exists so this endpoint can never delete an admin account even if the ID were tampered with. The delete is permanent: academic_records and recommendations for that user disappear too, via ON DELETE CASCADE in the schema (see Database section), not extra PHP code.

Views — public/*.php

Each page follows the same pattern: require its controller at the very top (which does all the thinking and can redirect before any HTML is sent), then print HTML using the variables the controller set. Logic here is limited to loops over already-fetched data and small display decisions.

index.phplanding page

No controller — just a session check that bounces a logged-in user straight to their dashboard, otherwise shows a static overview of the four "modules" with login/register links.

login.php / register.php

Thin wrappers around their controllers — a form, an error banner if $error is set, nothing else.

logout.php

No controller, no view — five lines: start the session, session_destroy(), redirect to login.

student_dashboard.phpthe score-submission form + result + history

Renders the five score inputs and interest dropdown (looped from SUBJECTS / ALLOWED_INTERESTS in constants.php, so the form can never drift from what the controller validates). If $result is set, shows the recommended pathway, the confidence bar (animated via a few lines of inline JS that widens it from 0% after a 100ms delay, for the fill animation), and the ranking table across all three pathways. Below that, partials/history_card.php renders every past submission. A hidden print-only header (.print-only) only becomes visible via the CSS @media print rules when the student clicks "Print or Save as PDF".

student_profile.php

Profile edit form pre-filled with the student's current name/email, plus optional password-change fields, wrapping student_profile_controller.php.

admin_dashboard.phpstats, retrain button, student roster, recent activity

The largest view in the app. Besides looping over the data the controller fetched, it has real client-side logic: a shared filterTable() JS function does live search + pathway filtering on both the student table and the recommendations table without a page reload or server round-trip — it just walks the table rows, checks each row's text against the search box and dropdown, and toggles display: none. filterStudents() and filterRecs() are thin wrappers that call it with each table's specific column indexes, and both run once on page load to set the initial "X of Y shown" count.

admin_student_report.php / admin_pathway_report.php

Printable report pages — a close button, a print button, and (for the student report) links to reset password / delete student. The pathway report's table is only rendered if $recommendations isn't empty; otherwise a "no students yet" message shows instead.

admin_reset_password.php / admin_delete_student.php

Small forms guarding a destructive/sensitive action. The delete page shows a plain-language warning ("this cannot be undone") naming exactly how many records will go with the account, before the confirmation field.

Partials — public/partials/

Reusable HTML fragments included by multiple pages, so a layout only has to be written once.

navbar.php

Reads three optional variables the including page sets beforehand — $navbar_admin, $navbar_greeting, $navbar_links — and renders the top bar accordingly. Every page includes this rather than repeating the markup.

history_card.php

Loops over $history (a set of joined recommendation + academic_record rows) and renders one card per past submission: pathway badge, confidence, timestamp, explanation text, and the five scores in a small table. Used by both student_dashboard.php (student viewing their own history) and admin_student_report.php (admin viewing any student's history) — same markup, different caller.

Assets — public/assets/

css/style.css

One stylesheet for the whole app — CSS custom properties for the color palette at the top, then cards, forms, buttons, tables, the confidence bar, and a dedicated @media print block that hides the navbar/buttons (.no-print) and reveals report-only headers (.print-only) when a page is printed or saved as PDF.

js/student.js

One listener: on submitting the recommendation form, if the form passes native HTML5 validation, disable the submit button and change its text to "Generating recommendation…" — feedback during the round trip to the ML API so the student doesn't wonder if their click registered or double-submit.

ML engine — ml/*.py

A separate Flask process on port 5001, called over HTTP by ml_client.php. Nothing here talks to PHP directly.

features.pysingle source of truth for feature order

Just a list — FEATURES = [math_score, english_score, science_score, humanities_score, creative_arts_score, interest_encoded]. Both train_model.py and api.py import this instead of each hardcoding their own copy, because the order here has to exactly match the order values are packed into a row at prediction time — a mismatch would silently feed the model the wrong column as "math score".

generate_data.pysynthetic training data generator

Produces the 3,000-row baseline dataset used until enough real student submissions accumulate. For each synthetic student: draws five subject scores from a normal distribution, then biases the relevant scores upward based on a randomly chosen interest (e.g. "technology" pushes math and science up), so the dataset has a genuine, learnable pattern rather than pure noise. The pathway label is computed with the same subject-strength + interest-bonus logic as build_explanation() in api.py — deliberately, so the label a human counselor would assign and the label the model explains its predictions against are the same reasoning. 8% of rows get their label randomly reassigned afterward ("label noise") so the dataset isn't trivially separable, which would make the reported accuracy meaningless.

train_model.pytrains and benchmarks all three models
  • load_data(df) — encodes the interest and pathway text columns to integers via LabelEncoder, returns X (features) / y (encoded pathway).
  • evaluate(...) — computes accuracy, macro precision/recall/F1, and prints a confusion matrix for one model.
  • train_and_save(df) — the main entry point. Splits 80/20 train/test (stratified so all three pathways are proportionally represented in both splits), trains Decision Tree, Random Forest, and KNN, evaluates each with both a held-out test set and 5-fold cross-validation, then only the Decision Tree is saved for production use (ml/model/decision_tree.joblib) — Random Forest and KNN's numbers are recorded in metrics.json purely as benchmarks proving the simpler, interpretable model is competitive. Separately, it fits a second KNN (n_neighbors=25) over scaled features across the whole population — this is the "collaborative filtering" component used at prediction time, distinct from the benchmark KNN above which is evaluated the same way as the other two models.

Called both directly (python ml/train_model.py, using the synthetic CSV) and by api.py's /retrain endpoint (passing in a DataFrame that blends real submissions with the synthetic baseline).

api.pythe Flask HTTP service — where predictions actually happen

Loads all five saved model files once at startup (the Decision Tree, its two encoders, the collaborative scaler, and the collaborative KNN) into module-level variables, so a prediction doesn't re-read files from disk on every request.

  • build_explanation(scores, interest, predicted_pathway) — generates the plain-English "why" text. Computes an average score per pathway area (STEM = avg of math+science, etc.), checks whether the strongest area matches the prediction and whether the stated interest maps to it, and composes a sentence from whichever of those are true. Falls back to "this pathway best balances your subject scores and interest" if neither condition holds, so the explanation is never empty.
  • predict() — the /predict route. Validates every field is present, the interest is a known category, and every score is a number between 0 and 100 (returning 400 with a specific message otherwise — this is what test_ml_api.py's rejection tests check). Runs the Decision Tree and the collaborative KNN independently, blends their probability outputs 70/30 (CONTENT_WEIGHT / COLLAB_WEIGHT), and picks the highest-probability pathway from the blend. Appends a second sentence to the explanation ("X% of students with a similar academic profile also pursued...") only when that peer agreement is 40% or higher — below that the collaborative signal is too weak to be worth stating as a supporting fact.
  • reload_models() — re-reads all five joblib files from disk; called after a retrain so the running Flask process picks up the newly trained model without needing a restart.
  • fetch_real_student_data() — a direct MySQL query (via pymysql, separate from PHP's PDO connection) joining academic_records → recommendations → pathways to pull every real submission and the pathway it was given.
  • retrain() — the /retrain route. Loads the synthetic baseline CSV, fetches real data, concatenates them (falling back to baseline-only if there are zero real submissions yet), calls train_and_save(), reloads the in-memory models, and returns the new metrics as JSON for the admin dashboard to display.
  • health() — trivial /health endpoint returning {"status": "ok"}, used by test_ml_api.py to confirm the Flask service is actually running before the other tests bother making requests.
model/metrics.json

Not code — the output of the last training run (accuracy/precision/recall/F1/cross-validation score per model). Overwritten every time train_and_save() runs, so the numbers here always reflect the currently-loaded model, not a fixed historical figure.

Database — database/schema.sql

Single source of truth for the schema — no migration files exist; this script is run once against a fresh MySQL instance to create everything, including the three fixed pathway rows.

TableHoldsKey columns
usersEvery account — students and the one adminuser_id PK, student_id (e.g. STU001, null for admin), role enum('student','admin'), password_hash
pathwaysThe 3 fixed CBC pathways — seeded by this script, never edited via the UIpathway_id PK, name, description
academic_recordsOne row per score submissionrecord_id PK, user_id FK, five score columns, interest
recommendationsOne row per AI result, tied to the submission that produced itrecommendation_id PK, user_id FK, record_id FK, pathway_id FK, confidence, explanation

How they relate (this is the answer to "explain the table relationships"):

RelationshipCardinalityOn delete
users → academic_records1 → manycascade
users → recommendations1 → manycascade
academic_records → recommendations1 → 1 in practice (one recommendation generated per submission)cascade
pathways → recommendations1 → manyno cascade

The two CASCADE rules on user_id are what makes admin_delete_student_controller.php's single DELETE FROM users statement sufficient — MySQL removes the student's academic records and recommendations automatically, no extra PHP delete statements needed. pathways deliberately has no cascade: it's reference data seeded once, never deleted through the app, so there's nothing to cascade from it. Two indexes (idx_academic_user, idx_recommendation_user) speed up the very common "all records/recommendations for this user" lookups used throughout the controllers.

Tests — tests/*.py

37 tests across three files, each testing a different layer.

test_unit_ml_logic.py9 tests — pure functions, no server or database needed

Imports build_explanation() and generate_student() directly and calls them in-process. Checks the explanation text cites the right reasons under different score/interest/pathway combinations and is never empty, and that synthetic student generation stays within valid ranges and produces the expected interest bias (technology-interested synthetic students average higher math scores).

test_ml_api.py14 tests — the Flask service over real HTTP, requires python ml/api.py running

Covers the happy path (/predict with a strong-STEM profile actually returns STEM, confidence is 0–100, ranking covers all three pathways) plus a deliberately long list of negative cases that must all return 400: missing fields, unknown interest, out-of-range scores (parametrized: too high, negative, just over/under the boundary), non-numeric score, null score, empty interest string, and a SQL-injection-style string in the interest field (this one exists specifically to confirm /predict never touches SQL at all, so there's nothing to inject into). One @pytest.mark.slow test actually calls /retrain end-to-end and checks the returned accuracy is above 70%.

test_web_flow.py14 tests — the PHP app over real HTTP, requires Apache + MySQL + Flask all running

The main test, test_register_login_submit_and_admin_visibility, is a full smoke test: register a fresh student, submit strong-STEM scores and confirm the recommendation comes back STEM, confirm a request with a forged CSRF token is rejected with 403, then log in as admin and confirm the new student's email shows up in the dashboard. Every other test targets one specific rejection path — wrong password, unregistered email, each individual password-strength rule, mismatched passwords, duplicate email, out-of-range scores, invalid interest, and a student trying to load the admin dashboard directly (expects 403). Test student accounts use student_{uuid}@test.com addresses and are not cleaned up automatically, so running this file repeatedly accumulates junk rows in the live database — DELETE FROM users WHERE email LIKE '%@test.com' clears them.
