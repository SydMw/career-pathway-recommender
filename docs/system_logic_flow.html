System logic flow

Every request the same way: page in public/ requires its controller, the controller does the thinking, the page renders what the controller decided. This traces that chain end to end, in the order things actually run.

Registration

New student creates an account

register.php→ require → register_controller.php→ requires → db.phpsession.phphelpers.php

Validates name/email/password (shared validate_password_strength()), checks the email isn't taken, hashes the password, generates a student ID like STU001 via generate_student_id(), inserts the row, then logs the student straight in.

$hash = password_hash($password, PASSWORD_DEFAULT);
$student_id = generate_student_id($pdo);
$pdo->prepare('INSERT INTO users (student_id, full_name, email, password_hash, role)
               VALUES (?, ?, ?, ?, ?)')->execute([$student_id, $full_name, $email, $hash, 'student']);

$_SESSION['user_id'] = (int) $pdo->lastInsertId();
$_SESSION['role'] = 'student';
header('Location: student_dashboard.php');

Login

Returning user signs in

login.php→ require → login_controller.php→ requires → db.phpsession.php

Looks up the email, then verifies the password. No lockout — wrong-password attempts aren't tracked or counted anywhere.

if (!$user) {
    $error = 'No account found with that email address...';
} elseif (!password_verify($password, $user['password_hash'])) {
    $error = 'Incorrect password. Please try again.';
} else {
    $_SESSION['user_id'] = $user['user_id'];
    $_SESSION['role'] = $user['role'];
    header('Location: ' . ($user['role'] === 'admin' ? 'admin_dashboard.php' : 'student_dashboard.php'));
}

Session guard — runs on every single request after login

src/config/session.php

Every controller requires this file before doing anything else: start (or resume) the PHP session, then provide the functions other controllers call explicitly — require_login() / require_role('admin') for access control, and require_csrf() guarding every POST.

if (session_status() === PHP_SESSION_NONE) {
    session_start();
}
4

Submit scores → get a recommendation

The core feature — FR4/FR5/FR6

student_dashboard.php→ require → student_controller.php→ requires → db.phpsession.phpml_client.phphelpers.phpconstants.php

Five subject scores + an interest are validated against SUBJECTS / ALLOWED_INTERESTS (constants.php), then it's a strict three-step sequence:

// 1. Save the raw submission
INSERT INTO academic_records (user_id, math_score, ..., interest) VALUES (...);
$record_id = $pdo->lastInsertId();

// 2. Ask the ML service for a prediction (HTTP call, see below)
$prediction = get_recommendation(['math_score' => ..., 'interest' => $interest]);

// 3. Save the result, linked back to the submission that produced it
INSERT INTO recommendations (user_id, record_id, pathway_id, confidence, explanation, model_used)
VALUES (?, ?, ?, ?, ?, ?);

get_recommendation() in ml_client.php is a plain cURL POST to the Flask service:

ml_client.php→ HTTP POST → ml/api.py :: /predict→ loads → ml/model/*.joblib

Inside Flask: the Decision Tree scores the student's own profile (content-based), a KNN model scores what similar peers in the training set ended up choosing (collaborative), and the two are blended 70/30.

content_proba = model.predict_proba(row)[0]                    # Decision Tree
collab_proba  = collab_knn.predict_proba(row_scaled)[0]         # peer KNN
blended = 0.7 * content_proba + 0.3 * collab_proba
predicted_pathway = label_encoder.inverse_transform([argmax(blended)])[0]

Flask returns pathway + confidence + a plain-English explanation (which subject area drove it, whether stated interest matched) back over HTTP as JSON; PHP decodes it and runs step 3 above.

6

Admin dashboard + retrain

FR7 (visibility) / FR8 (retraining)

admin_dashboard.php→ require → admin_controller.php→ requires → db.phpsession.phpml_client.phphelpers.php

On GET: five queries build the whole page — totals, breakdown by pathway, 50 most recent recommendations, the full student roster (left-joined so students with zero submissions still show up), and a PHP-side pass to find each student's latest pathway (avoids a fragile correlated subquery).

On POST (retrain): calls trigger_retrain() → Flask /retrain, which blends the synthetic baseline CSV with real accumulated submissions, re-trains all three models, and hot-swaps them into the running Flask process.

# ml/api.py :: /retrain
baseline = pd.read_csv("ml/data/students.csv")
real_data = fetch_real_student_data()          # academic_records + recommendations, real users only
combined = pd.concat([baseline, real_data])
results = train_and_save(combined)             # ml/train_model.py
reload_models()                                # swaps the in-memory model without restarting Flask
7

Delete a student

Permanent — no trash, no restore

admin_delete_student.php→ admin_delete_student_controller.php

Delete requires typing the student's full name to confirm, then removes the row outright:

DELETE FROM users WHERE user_id = ? AND role = ?

The foreign keys on academic_records and recommendations are ON DELETE CASCADE, so their submissions and recommendations are removed automatically along with the account. There's no soft-delete flag anywhere in the schema — once confirmed, this cannot be undone.

Printable reports

admin_student_report.php· admin_pathway_report.php

Two read-only views: one student's full recommendation history, or one pathway's aggregate stats (average scores + confidence across everyone recommended into it). Both are plain GET pages with an ?id= / ?pathway= query param, no state changes.

10

Logout

logout.php

One line — destroys the session and redirects to login.php. There is no automatic inactivity timeout; a session stays valid until the user logs out or closes the browser (session cookie).

Training pipeline — where the models in step 4 and step 6 actually come from

ml/features.py + ml/train_model.py

train_model.pyimports features.py· api.pyimports features.py

features.py is nothing but a six-item list — but it's the thing that keeps training and prediction from silently drifting apart. Both files build a row in exactly this column order; if one ever reordered it independently, the model would still run, just score every prediction against the wrong columns.

FEATURES = ["math_score", "english_score", "science_score",
            "humanities_score", "creative_arts_score", "interest_encoded"]

load_data() turns the raw CSV (or a blended DataFrame, when called from /retrain) into model-ready arrays: interests and pathways are text, so each gets its own LabelEncoder — the same two encoder objects get saved and reused at prediction time, so "technology" always maps to the same integer.

interest_encoder = LabelEncoder()
df["interest_encoded"] = interest_encoder.fit_transform(df["interest"])
label_encoder = LabelEncoder()
df["pathway_encoded"] = label_encoder.fit_transform(df["pathway"])
X = df[FEATURES]           # the 6 columns, in the exact order above
y = df["pathway_encoded"]

train_and_save() does an 80/20 stratified split, then trains three classifiers side by side purely to compare them — accuracy, macro precision/recall/F1, and 5-fold cross-validation for each, written to ml/model/metrics.json (the file that shows as changed after every retrain).

dt  = DecisionTreeClassifier(max_depth=6, min_samples_leaf=10, random_state=42)
rf  = RandomForestClassifier(n_estimators=200, max_depth=10, random_state=42)
knn = KNeighborsClassifier(n_neighbors=9)

for name, model in [("Decision Tree", dt), ("Random Forest", rf), ("KNN", knn)]:
    model.fit(X_train, y_train)
    acc, report = evaluate(name, model, X_test, y_test, label_encoder)
    cv_scores = cross_val_score(model, X, y, cv=5)

Only the Decision Tree gets deployed — shallow on purpose (max_depth=6) so its decision path stays readable enough to explain to a student, per FR5. The Random Forest and this first KNN never leave metrics.json; they exist purely as a "how would a fancier model have done" comparison for the report.

joblib.dump(dt, "ml/model/decision_tree.joblib")
joblib.dump(interest_encoder, "ml/model/interest_encoder.joblib")
joblib.dump(label_encoder, "ml/model/label_encoder.joblib")

A second, separate KNN is then built — this is the one that actually runs live in every prediction (the "collaborative" half of the blend back in step 4). It's trained on the full dataset rather than the 80% training split, and on StandardScaler-scaled features rather than raw scores, so no single subject's scale dominates distance calculations. Don't confuse it with the benchmark KNN above — same algorithm, different job, different neighbor count.

scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)
collab_knn = KNeighborsClassifier(n_neighbors=25)   # "the 25 most similar students"
collab_knn.fit(X_scaled, y)
joblib.dump(scaler, "ml/model/collab_scaler.joblib")
joblib.dump(collab_knn, "ml/model/collab_knn.joblib")

Two ways this function gets called: directly as python ml/train_model.py (the if __name__ == "__main__" guard, no arguments, always trains on the synthetic CSV) — or from inside Flask's /retrain route in step 6, passed the synthetic-plus-real blended DataFrame instead. Same function, same five saved .joblib files, different input.

Shared config layer

Every controller pulls from these — nothing above duplicates this logic, it's required fresh each time.

FileResponsibility
config/db.phpOpens the one PDO connection ($pdo), used by every controller that touches MySQL.
config/session.phpSession start, require_login(), require_role(), CSRF token + verification.
config/helpers.phpvalidate_password_strength(), generate_student_id(), require_student_by_id() — small cross-cutting logic shared by 2+ controllers.
config/ml_client.phpget_recommendation() and trigger_retrain() — the only two places PHP ever talks to Flask, both plain cURL.
config/constants.phpSUBJECTS and ALLOWED_INTERESTS — one source of truth shared by the form (student_dashboard.php) and its validation (student_controller.php).
ml/api.py is the Flask process itself (port 5001, started by start_ml.bat) — step 11 covers what actually produces the .joblib files it loads.
